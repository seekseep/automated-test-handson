# レクチャー: 良いテストの性質

> 解説: [docs/03_basic.md](../../../docs/03_basic.md)

## このレクチャーのゴール

**速い・決定的・1つを検証・名前が仕様** という4つの観点で、自分のテストを見直せるようにする。

## 前提

[../03-assertions](../03-assertions) を済ませていること（アサーションを使い分けられる）。

## 題材

- [src/order.js](./src/order.js) … 現在時刻を**引数で受け取る** `isSameDay`
- [test/order.test.js](./test/order.test.js) … 4つの性質を体現したテスト

## 4つの性質

| 性質 | 意味 | このレクチャーでの体現 |
|---|---|---|
| 速い | すぐ実行でき、何度も回せる | 純粋関数だけを呼ぶ（DB・通信なし） |
| 決定的 | 何度実行しても同じ結果 | 時刻を引数で渡す（`new Date()` を関数内で呼ばない） |
| 1つを検証 | 1テスト1観点。落ちたら原因が一目で分かる | 「当日」と「日またぎ」を別テストに分ける |
| 名前が仕様 | 名前だけで何を保証しているか読める | `'注文時刻と現在時刻が同じ日付なら当日と判定する'` |

## 手順

### 1. 決定的にする

[src/order.js](./src/order.js) は現在時刻を引数 `now` で受け取る。
もし関数内で `new Date()` を呼ぶと、実行する日によって結果が変わり「ある日突然落ちる」テストになる。
外部依存（時刻・乱数・通信）は**引数で注入**して切り離す。

### 2. 1テスト1検証に分割する

[test/order.test.js](./test/order.test.js) は観点ごとにテストを分けている。
1つのテストで欲張らないと、落ちたとき原因がすぐ分かる。

### 3. テスト名を仕様文に直す

`'test1'` のような名前ではなく、保証している仕様をそのまま日本語で書く。
テスト名の一覧が、そのまま仕様の一覧になる。

### 4. 実行する

```sh
node --test
```

## まとめ

テストは**仕様を実行可能な形で書いたもの**。
速く・決定的で・1観点ずつ・名前が仕様になっていると、壊れたときに最大限助けてくれる。
