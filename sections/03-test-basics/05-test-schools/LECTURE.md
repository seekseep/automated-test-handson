# レクチャー: 古典学派とロンドン学派

> 解説: [docs/03_basic.md](../../../docs/03_basic.md)

## このレクチャーのゴール

自動テストには大きく2つの流派があることを知り、**同じ題材を両方のやり方でテストして違いを体感する**。

- **古典学派（Classical / Detroit）**: 協力オブジェクトは本物を使い、**結果（状態・戻り値）**を検証する。
- **ロンドン学派（London / mockist）**: 協力オブジェクトはモックに差し替え、**相互作用（どう呼んだか）**を検証する。

## 前提

[../04-good-tests](../04-good-tests) を済ませていること（良いテストの4性質を理解している）。

## 用語

- **テスト対象（SUT: System Under Test）**: いま検証したい関数・モジュール。ここでは `placeOrder`。
- **協力オブジェクト（collaborator）**: SUT が依存して呼び出す相手。ここでは在庫 `inventory`。
- **モック / スタブ**: 協力オブジェクトの**偽物**。呼ばれ方を記録したり、決まった値や例外を返す。

## 題材

- [src/order.js](./src/order.js) … 在庫を引き当てて合計を返す `placeOrder`（テスト対象）
- [src/inventory.js](./src/inventory.js) … 協力オブジェクト `createInventory`（本物の在庫）
- [test/classical.test.js](./test/classical.test.js) … 古典学派のテスト（本物の在庫＋状態検証）
- [test/london.test.js](./test/london.test.js) … ロンドン学派のテスト（モック＋相互作用検証）

## 2つの流派の違い

| 観点 | 古典学派（Classical） | ロンドン学派（London） |
|---|---|---|
| 協力オブジェクト | できるだけ**本物**を使う | **モック**に差し替える |
| 何を検証するか | **結果**（最終状態・戻り値） | **相互作用**（何をどう呼んだか） |
| 隔離するもの | テスト同士（共有依存だけモック） | テスト対象（協力相手は全部モック） |
| リファクタ耐性 | 強い（内部の呼び方を変えても緑） | 弱い（呼び方を変えると赤になりやすい） |
| バグの原因究明 | 広め（協力相手のバグも巻き込む） | 狭い（対象だけを切り出せる） |

どちらが「正しい」わけではない。**何を検証したいか**で選ぶ。

## 手順

### 1. 古典学派で書く

[test/classical.test.js](./test/classical.test.js) では `createInventory` の**本物**を渡す。
注文後に「合計金額」と「残った在庫」という**状態**を確かめる。
`placeOrder` が内部で `reserve` を何回呼んだかには一切触れていない点に注目。

### 2. ロンドン学派で書く

[test/london.test.js](./test/london.test.js) では `node:test` の `mock.fn()` で偽の `reserve` を作って渡す。
検証するのは「`reserve` が `['apple', 3]` で呼ばれたか」という**相互作用**。
本物の在庫は一切登場しない。

### 3. 両方まとめて実行する

```sh
node --test
```

両方のテストファイルが緑になることを確認する。
同じ `placeOrder` を、**まったく違う着眼点**で検証できていることを味わう。

## 使い分けの指針

- 迷ったら**古典学派を基本**にする（リファクタに強く、壊れにくい）。
- 本物を用意しづらい依存（外部API・メール送信・決済など、**呼んだこと自体が重要な副作用**）は、ロンドン学派的にモックで「呼ばれ方」を検証すると相性が良い。
- モックを増やすほどテストは実装に密着し、壊れやすくなる。**モックは必要な境界だけに留める**。

## まとめ

- 古典学派は **結果**を、ロンドン学派は **相互作用**を見る。
- 同じコードでも、どちらの流派かで「何を検証するテストか」が変わる。
- 基本は古典学派。副作用の有無で検証したいものが変わるとき、ロンドン学派を選ぶ。
